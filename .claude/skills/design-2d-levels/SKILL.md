# 2Dゲームのレベル設計

ゲームが目指す体験から逆算し、プレイヤーが理解、判断、上達できる
2Dレベルを設計、実装、評価する。

## 必須コンテキスト

作業開始時に `.agents/skills/design-2d-levels/references/game-design-hints.md` を読む。

次に、対象リポジトリに存在する仕様書、設計書、進捗記録、既存レベル定義、
ゲームプレイ設定、テストを探して読む。ファイル名を決め打ちしない。

文書間で矛盾があれば、現在の実装と明示された最新仕様を優先する。
面白さに関する提案は事実ではなく、検証可能な仮説として扱う。

## 設計手順

### 1. 一文の目的を決める

ゲーム全体が目指す感覚を確認し、そのレベルでプレイヤーに
学習、判断、発見させる内容を一つに絞る。

良い例:

- 基本移動を安全な空間で自力発見させる。
- 正面突破と遠回りの安全策を選ばせる。
- 既知の能力を意外な地形で使わせる。

「敵を増やして難しくする」のような、体験を示さない目的は使わない。

### 2. 目標体験から配置を逆算する

Mechanics の追加より、既存ルールから望む Dynamics が生じる配置を優先する。

- 少なくとも一つ、初見でも進行できる安全解を置く。
- 可能なら、理解や精度に報いる高得点解を別に置く。
- 単一の入力や能力の反復だけが常に最適にならないようにする。
- 敵や危険物を単なる通行禁止ではなく、判断を作る境界として使う。
- 成功が偶然ではなく、理解と技能によって再現可能になるようにする。
- 作者の想定外でも面白い解は許容する。

### 3. 教える順序を守る

新要素は安全に単独で試させ、成功後に既知要素と組み合わせる。
一つのレベルまたは連続するレベル群を、次の順で構成する。

1. 導入: 安全で単純な条件。
2. 展開: 同じ要素を少し難しい条件で使用。
3. 転換: 反転、複合、意外な使用法。
4. 結論: 学習内容を短く試す。

新要素を二つ以上同時に初登場させない。

### 4. 失敗を情報にする

失敗後にプレイヤーが次の一発を修正できるか確認する。

- 入力、位置、タイミング、ルール理解のどれが原因か読み取れる。
- 地形、敵、危険物の判定が見た目と一致する。
- 補助表示や直前の結果が答えを見せすぎず、比較材料になる。
- 再挑戦までの待ち時間が短い。

精密さを要求する前に、因果が理解可能で再現可能かを優先する。

### 5. 数値と到達可能性を検証する

配置を実装する場合は、ゲームに適した手段で代表的な攻略経路を確認する。

- 安全解と上級者向け解が実際に到達可能か。
- 資源、制限時間、チェックポイントが想定解と一致するか。
- 複数の操作や能力に異なる用途があるか。
- 危険物が必須経路を理不尽に塞いでいないか。
- スコアや評価基準が想定攻略から説明可能か。
- 必要な再現性が保たれているか。

新規または変更したレベルには、到達可能性、勝敗、得点、進行のうち
変更リスクに合う自動テストを追加する。

### 6. 観察可能な検証仮説を付ける

「面白いはず」で終わらせず、プレイテストで観察できる指標を決める。

- 最初の意味ある行動までの時間。
- 使用した経路、能力、資源の分布。
- 同じ失敗を繰り返した回数。
- 失敗後に行動を修正した割合。
- 安全解と上級者向け解の選択割合。
- クリア後に別解や追加目標を狙って再挑戦した割合。

比較テストでは一度に複数条件を変えない。

## 出力形式

レベル案またはレビューでは、最低限次を示す。

```markdown
## レベル名

- 目的:
- プレイヤーに体験させたい感覚:
- 新規要素:
- 配置の意図:
- 安全解:
- 上級者向け解:
- 失敗から得られる情報:
- 資源・制限・評価基準:
- 想定外解の許容範囲:
- プレイテスト仮説:
```

実装時は、変更したレベル定義、検証した代表経路、テスト結果、
未検証のリスクを簡潔に報告する。リポジトリに進捗記録があれば更新する。

## 避ける設計

- 敵数、速度、危険物、耐久値だけで難度を上げる。
- 正解が一つの狭いタイミングしかなく、失敗理由を読めない。
- 補助表示に従うだけで解ける。
- クリア解と最高得点解が常に同一になる。
- 新しい敵やギミックで、未検証のコアの弱さを覆う。
- 演出の長さによって再試行のテンポを損なう。
